# Bookmark Nav 长期路线规划

## 1. 产品定位

Bookmark Nav 的长期方向不是只做一个“好看的新标签页”，而是逐步演进为一个本地优先、可搜索、可整理、可备份的浏览器书签工作台。

一句话定位：

> 一个本地优先、最小权限、面向长期使用的浏览器书签工作台。

这个定位里有三个核心约束：

- 本地优先：书签、设置、历史记录和整理结果默认保存在本地，用户不需要把数据交给远端服务。
- 最小权限：扩展只申请完成书签管理所必需的权限，避免无关的页面注入和网络权限。
- 可恢复：任何会修改浏览器书签的操作，都应该有确认、备份、撤销或操作历史。

Bookmark Nav 最有价值的入口是“新标签页”。用户每天都会打开新标签页，因此它天然适合承载书签搜索、常用入口、最近访问、临时收藏和工作区切换。

## 2. 目标用户

### 2.1 主要用户

- 长期积累大量浏览器书签的人。
- 开发者、设计师、内容创作者、研究型用户等高频资料检索用户。
- 需要在多个主题、项目、资料源之间快速切换的人。
- 不希望把书签托管到第三方在线导航服务的人。

### 2.2 典型痛点

- 书签越来越多，靠文件夹层级已经很难找到。
- 同一个链接被重复收藏在多个文件夹。
- 很多书签已经失效，但用户不知道。
- 旧书签标题混乱，无法看出真实用途。
- 浏览器自带书签管理器操作成本高，不适合批量整理。
- 用户希望有一个高效的新标签页入口，但不想牺牲隐私和权限安全。

## 3. 产品原则

### 3.1 默认不破坏

删除、移动、批量编辑、清理历史等操作必须提供明确确认。对于批量操作，优先提供预览和撤销。

### 3.2 建议先于执行

智能整理、去重、重命名、分类等能力应先提供建议，再让用户选择是否应用。尤其是 AI 相关能力，不应默认直接修改书签树。

### 3.3 搜索优先

书签数量越多，搜索越重要。产品体验应优先保障“输入几个字符后快速找到目标”，而不是只依赖视觉分组。

### 3.4 权限透明

README、扩展商店说明和应用内设置都应清楚解释权限用途。当前 `bookmarks` 权限是产品信任基础，应尽量长期保持最小权限策略。

### 3.5 数据可迁移

设置、历史记录、标签、工作区和整理结果都应支持导出/导入。用户不应被锁死在扩展里。

### 3.6 尊重现有 MVP 边界

当前 `spec.md` 已经明确了一组低风险 MVP 边界：不注入页面、不需要后端、不默认请求网络权限、导出/导入不读写浏览器书签树、书签移动通过选择目标文件夹完成而不是拖拽排序。路线规划中的长期能力应以这些边界为基础逐步演进，不能把高风险能力提前塞进稳定发布版。

## 4. 阶段路线

## 阶段一：稳定发布版

目标：把 Bookmark Nav 打磨成可以公开发布、可长期使用的新标签页扩展。

### 4.1 关键任务

- 完成 Chrome MV3、Edge、Firefox 的构建和安装说明。
- 上架 Chrome Web Store、Edge Add-ons，条件成熟后上架 Firefox Add-ons。
- 补齐仓库 `description`、`topics`、`license`、release 截图和安装包说明。
- 在 README 中增加更偏用户的说明：
  - 产品截图。
  - 安装方式。
  - 权限解释。
  - 常见问题。
  - 与浏览器原生书签管理器的区别。
- 完善搜索排序：
  - 标题命中优先。
  - 域名命中优先。
  - 文件夹路径命中。
  - 最近打开和常用书签加权。
  - 排除词、站点限定和文件夹限定语法继续保留。
- 加强测试覆盖：
  - 搜索排序。
  - 书签树解析。
  - 删除、移动、批量操作。
  - 设置导入/导出。
  - manifest 权限检查。

### 4.2 验收标准

- 用户可以通过 README 或 Release 独立完成安装。
- 新标签页核心功能稳定，无明显空白状态和异常状态。
- 扩展只声明必要权限。
- 删除、批量删除、移动等危险操作都有确认。
- `pnpm verify` 能稳定通过。

## 阶段二：书签健康检查

目标：帮助用户理解自己的书签状况，而不仅仅是展示书签。

### 4.3 关键任务

- 增加“书签整理报告”：
  - 重复链接。
  - 相似链接。
  - 空文件夹。
  - 长期未访问书签。
  - 可能失效的链接。
  - 标题过短或无意义的书签。
- 增加重复书签处理：
  - 展示重复项所在文件夹。
  - 支持保留其中一个，删除其他重复项。
  - 支持按最近访问、文件夹优先级、创建时间辅助选择。
- 增加失效链接检测：
  - 初期可以只做用户手动触发。
  - 避免默认后台大规模请求外部 URL。
  - 检测前说明可能产生网络请求。
- 增加清理前备份：
  - 批量删除或移动前生成快照。
  - 支持导出清理报告。

### 4.4 验收标准

- 用户能在不修改书签的情况下先看到整理建议。
- 所有批量清理操作都有预览。
- 整理报告不会默认向外部发送用户书签数据。

## 阶段三：真正的书签管理器

目标：从“浏览书签”升级为“高效整理书签”。

### 4.5 关键任务

- 继续使用“选择目标文件夹”的移动方式作为主要交互，避免早期拖拽造成误移动。
- 在操作历史、撤销和批量操作备份成熟后，再评估是否支持拖拽移动书签；文件夹拖拽排序应作为更晚期能力单独评估。
- 支持批量改名、批量移动、批量删除。
- 支持操作历史：
  - 删除历史。
  - 移动历史。
  - 批量操作记录。
  - 撤销入口。
- 增加标签系统：
  - 标签独立于浏览器原生文件夹。
  - 一个书签可以有多个标签。
  - 标签数据保存在本地设置中。
  - 支持按标签过滤和搜索。
- 增加“收件箱”：
  - 快速保存当前页面到临时区。
  - 稍后批量整理到正式文件夹。
- 增加文件夹管理：
  - 空文件夹清理。
  - 文件夹合并建议。
  - 文件夹重命名建议。

### 4.6 验收标准

- 用户可以完成主要书签整理动作，不再频繁打开浏览器原生书签管理器。
- 批量操作可以被追踪和恢复。
- 标签系统不会破坏原生书签结构。

## 阶段四：工作区与命令面板

目标：把新标签页变成日常工作入口。

当前进展：工作区第一版已作为阶段三后的体验增强提前完成，支持本地工作区、快速创建、组合文件夹/标签/搜索词和导出/导入。阶段四后续重点应转向键盘效率入口，而不是继续增加重型管理功能。

### 4.7 关键任务

- 完善工作区：
  - 按开发、写作、学习、项目等场景组织入口。
  - 工作区可以绑定文件夹、标签、常用搜索和外部搜索源。
  - 支持快速切换工作区。
- 增加命令面板：
  - 打开书签。
  - 搜索书签。
  - 搜索网页。
  - 打开最近访问。
  - 切换文件夹、标签和工作区。
  - 第一版不加入删除、移动、批量编辑等危险管理动作，避免键盘误操作直接修改真实书签。
- 支持自定义搜索引擎：
  - Google、Bing、DuckDuckGo、百度继续保留。
  - 用户可配置 GitHub、npm、MDN、Stack Overflow、知乎等搜索源。
  - 支持快捷前缀，如 `gh react`、`npm vite`、`mdn flex`。
- 增加项目入口聚合：
  - 用户可把 GitHub、文档、部署地址、监控页、笔记链接聚合成一个项目组。

### 4.8 验收标准

- 用户打开新标签页后，可以通过键盘完成大部分跳转。
- 工作区切换能明显减少书签层级查找成本。
- 命令面板不会替代现有 UI，而是成为高频用户的效率入口。

## 阶段五：AI 辅助整理

目标：引入 AI，但保持低风险和可控。

### 4.9 适合做的 AI 能力

- 自动生成书签分类建议。
- 总结某个文件夹下链接的主题。
- 给标题较差的书签生成更清楚的名称。
- 根据搜索意图推荐相关书签。
- 识别同一项目相关链接。
- 生成“最近常用资料入口”摘要。
- 为工作区推荐候选书签。

### 4.10 不建议早期做的 AI 能力

- AI 自动修改书签树。
- AI 默认删除重复书签。
- AI 默认批量重命名。
- 默认把全部书签上传到远端模型。

### 4.11 AI 功能原则

- AI 只做建议，用户确认后才执行。
- AI 请求前明确告知会发送哪些数据。
- 支持用户选择本地模型、远端模型或完全关闭 AI。
- 对发送数据做最小化处理，只发送必要标题、URL 域名、文件夹路径或摘要。

### 4.12 验收标准

- 用户可以清楚知道 AI 使用了哪些数据。
- AI 建议可以被逐项采纳或拒绝。
- 关闭 AI 后，核心书签管理功能仍然完整可用。

## 阶段六：同步、备份与生态

目标：让用户不会被单设备绑定，同时不牺牲本地优先原则。

### 4.13 关键任务

- 完善导出/导入：
  - 设置。
  - 标签。
  - 工作区。
  - 常用/最近历史。
  - 操作历史。
- 支持备份：
  - 手动备份。
  - 定期提醒备份。
  - 批量操作前自动生成备份。
  - 备份优先覆盖扩展本地增强数据和操作记录；如未来要备份浏览器书签树，应作为独立功能说明清楚，不混入现有设置导出。
- 支持可选同步：
  - WebDAV。
  - GitHub Gist。
  - iCloud/Dropbox 等本地同步目录。
- 支持外部工具互通：
  - Obsidian。
  - Notion。
  - Readwise。
  - Raindrop。
- 支持只读分享页：
  - 将某个文件夹或工作区导出为公开导航页。
  - 分享前明确告知会暴露哪些链接。

### 4.14 验收标准

- 用户可以在新设备恢复主要配置。
- 同步是可选能力，不影响本地单机使用。
- 分享功能不会默认公开私人书签。

## 5. 功能优先级

### 5.1 高优先级

- 仓库展示补全：description、topics、license、README 截图。
- 商店发布准备。
- 搜索排序优化。
- 重复书签检测。
- 批量操作预览。
- 操作历史和撤销。
- 导出/导入。

### 5.2 中优先级

- 标签系统。
- 工作区。
- 命令面板。
- 自定义搜索引擎。
- 失效链接检测。
- 文件夹整理建议。

### 5.3 低优先级

- AI 辅助整理。
- 多端同步。
- 外部工具集成。
- 公开分享页。
- 主题市场或插件系统。

## 6. 技术路线建议

### 6.1 数据模型

建议把数据分为三类：

- 浏览器原生书签数据：来自 `chrome.bookmarks`，修改会影响真实书签。
- 扩展本地增强数据：标签、工作区、最近访问、常用记录、UI 设置。
- 操作与备份数据：批量操作记录、撤销栈、导出快照。

这样可以避免把增强功能强行写入浏览器书签结构。

### 6.2 搜索能力

搜索可以逐步演进：

- 第一层：关键词、排除词、站点限定、文件夹限定。
- 第二层：拼音、模糊匹配、域名权重、最近访问权重。
- 第三层：标签、工作区、语义相关性。
- 第四层：AI 查询意图辅助。

### 6.3 安全与权限

长期应尽量保持：

- 不注入页面。
- 不默认请求网络权限。
- 不默认上传书签。
- 不默认修改用户数据。

如果未来确实需要网络能力，比如失效链接检测、同步、AI，应拆分成用户主动启用的功能，并在 UI 和文档里解释原因。

### 6.4 测试策略

重点覆盖以下模块：

- 书签树解析。
- 搜索排序。
- 批量操作。
- 导入/导出。
- 设置归一化。
- 权限 manifest 检查。
- Firefox 与 Chrome manifest 差异。

对于会修改书签的能力，建议增加 mock bookmark tree 的单元测试和集成测试。

## 7. 商店发布建议

### 7.1 发布材料

- 一句话描述：本地优先的浏览器书签导航与管理工具。
- 三到五张截图：
  - 新标签页总览。
  - 搜索结果。
  - 文件夹侧边栏。
  - 批量管理。
  - 设置页。
- 权限说明：
  - `bookmarks`：读取和管理浏览器书签。
  - 无页面注入。
  - 无默认网络上传。
- README 中放置安装包和商店链接。

### 7.2 版本节奏

建议保持小步发布：

- `1.0.x`：稳定性、文档、商店发布修复。
- `1.1.x`：搜索优化和整理报告。
- `1.2.x`：批量管理和操作历史。
- `1.3.x`：标签和工作区。
- `1.4.x`：命令面板和自定义搜索。
- `2.0.x`：如果引入 AI 或同步等会改变权限和数据策略的大功能，再考虑 major 版本。

## 8. 风险与取舍

### 8.1 权限风险

一旦加入网络、页面注入或同步权限，用户信任成本会明显提高。应尽量把这些能力做成可选模块。

### 8.2 数据破坏风险

书签是用户长期资产。移动、删除、合并、重命名都可能造成损失。批量操作必须有预览、备份和撤销。

### 8.3 功能膨胀风险

不要过早做复杂仪表盘、主题市场、AI 聊天助手或完整知识库。产品的核心仍然是书签入口和书签整理。

### 8.4 跨浏览器维护成本

Chrome、Edge、Firefox 对 manifest、权限和扩展 API 的要求不同。每新增一项能力，都应评估跨浏览器成本。

## 10. 总结

Bookmark Nav 最值得长期投入的方向，是把它从一个新标签页扩展推进为一个可信赖的书签工作台。

近期不要急着做 AI 和同步。更高价值的顺序是：

1. 发布稳定版。
2. 做好搜索。
3. 做好整理报告。
4. 做好批量操作和撤销。
5. 做好标签与工作区。
6. 做好命令面板和自定义搜索。
7. 再引入 AI 辅助和可选同步。

只要坚持本地优先、最小权限、可恢复这三个原则，Bookmark Nav 就能和普通在线导航页、书签同步服务、AI 工具拉开差异。
